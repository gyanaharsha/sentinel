How the check stays private

The naive way to check a password against a breach list is to send the password to a server — which is itself a leak. This is the workaround I found genuinely elegant when I first studied it: k-anonymity, the same range-query protocol password managers like 1Password use under the hood.

password → 40 hex chars

Hash locally

Your browser computes the SHA-1 fingerprint of the password. This happens on your machine, in this page's code — the password itself never exists anywhere else.

send: first 5 only

Ask by prefix

Only the first 5 characters of the fingerprint are sent to the Have I Been Pwned database. That prefix matches roughly 800 different leaked hashes — the server cannot tell which one, if any, is yours.

match happens here

Compare locally

The server returns every suffix in that range. Your browser checks — privately, on your device — whether the remaining 35 characters appear in the list.

Instead of asking the librarian "do you have my diary?" — you ask for everything on shelf 5B and look through the shelf yourself. The librarian never learns which book you were after.
